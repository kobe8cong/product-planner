# Product Planner Skill - 深度产品分析

## 📋 项目基本信息

**项目名称：** Product Planner (Claude Code Plugin)  
**定位：** 为开发者提供系统化产品规划能力的 Claude Code skill  
**目标用户：** 独立开发者、开源项目维护者、小团队技术创始人  
**核心价值：** 让技术人员拥有产品经理的思维框架和分析工具  

---

## 🎯 Part 1: 问题定义与用户洞察

### 1.1 我们要解决什么问题？

**核心问题：**
开发者擅长写代码，但常常：
- ❌ 凭感觉做产品，缺乏系统方法论
- ❌ 直接开始编码，跳过需求分析和市场研究
- ❌ 功能堆砌，不知道什么是 MVP
- ❌ 不了解目标用户真正的痛点
- ❌ 项目做到一半发现方向错了
- ❌ 开源项目无人问津（因为没解决真实需求）

**为什么这个问题重要？**
- 💰 浪费时间开发无人使用的功能
- 😞 项目失败率高，打击积极性
- 📉 开源项目难以获得 stars 和贡献者
- 🚫 错过真正的市场机会

**当前解决方案的不足：**
1. **产品管理书籍/课程** - 理论多，落地难，不适合技术人员
2. **人工咨询** - 成本高，不可扩展，反馈周期长
3. **模板工具** - 死板，缺乏针对性分析
4. **Claude Code 现有 plugins** - 侧重技术实现，缺产品前置思考

### 1.2 目标用户画像

#### 主要用户 Persona 1: 独立开发者 Alex
- **背景：** 5年开发经验，想做 side project
- **痛点：** 
  - 有很多想法但不知道哪个值得做
  - 做了几个项目都没人用
  - 不知道如何定义 MVP
- **目标：** 创建一个有真实用户的产品
- **使用场景：** 周末想开始新项目，需要快速验证想法

#### 主要用户 Persona 2: 开源维护者 Sarah
- **背景：** 开源项目维护者，想让项目更受欢迎
- **痛点：**
  - 不清楚用户真正需要什么功能
  - 功能路线图混乱
  - 难以吸引贡献者
- **目标：** 增加项目 stars，建立活跃社区
- **使用场景：** 规划下一个版本功能，优先级排序

#### 次要用户 Persona 3: 技术创始人 Mike
- **背景：** 创业公司 CTO，小团队（3-5人）
- **痛点：**
  - 没有专职 PM，技术和产品都要管
  - 需要快速做产品决策
  - 资源有限，不能试错
- **目标：** 用有限资源做出 PMF 产品
- **使用场景：** 规划新功能，评估投入产出比

### 1.3 用户旅程分析

**现状（without Product Planner）：**
1. 💡 产生想法 → 
2. 🤔 自己简单思考 → 
3. 💻 直接开始编码 → 
4. 😰 开发中发现问题 → 
5. 🔄 反复修改返工 → 
6. 😞 发布后无人问津

**痛点出现在：**
- 步骤 2: 缺乏系统分析框架
- 步骤 3: 没有验证需求就开始
- 步骤 4: 架构没想清楚
- 步骤 6: 没有 GTM 策略

**理想状态（with Product Planner）：**
1. 💡 产生想法 → 
2. 🔍 **使用 Product Planner 深度分析** → 
3. 📊 **获得数据驱动的洞察** → 
4. ✅ **明确 MVP 和路线图** → 
5. 💻 信心满满地开发 → 
6. 🚀 **按 GTM 策略发布** → 
7. 🎉 获得用户和反馈

## 🔍 Part 2: 市场与竞品分析

### 2.1 市场规模与趋势

**目标市场：**
- **一级市场：** Claude Code 用户（开发者生态）
- **二级市场：** AI 辅助开发工具用户
- **三级市场：** 所有技术创业者、独立开发者

**市场规模估算：**
- Claude Code 用户：数万级别（快速增长中）
- GitHub 活跃开发者：1亿+
- 独立开发者社区：Indie Hackers (100万+), Product Hunt (500万+)

**市场趋势：**
- ✅ AI 辅助开发工具快速普及
- ✅ 独立开发者数量持续增长
- ✅ 产品思维在技术社区受重视
- ✅ No-code/Low-code 工具兴起（证明需求）

### 2.2 竞品分析

#### 竞品 1: Product Hunt Ship
**优势：**
- 成熟的产品发布平台
- 社区反馈机制
**劣势：**
- 不是 AI 工具，无法自动分析
- 专注发布阶段，不管规划
- 独立于开发工作流

#### 竞品 2: Notion/Coda 产品模板
**优势：**
- 灵活的文档工具
- 模板丰富
**劣势：**
- 静态模板，无智能分析
- 需要手动填写，费时费力
- 与代码开发割裂

#### 竞品 3: Linear/Jira
**优势：**
- 强大的项目管理
- 团队协作功能
**劣势：**
- 侧重执行，非产品规划
- 重量级工具，个人开发者不适合
- 无产品分析能力

#### 竞品 4: ChatGPT/Claude 对话
**优势：**
- 灵活，可以问任何问题
**劣势：**
- 非结构化，缺乏框架
- 没有集成到开发工作流
- 分析不够深入和系统
- 无持久化，无法复用

### 2.3 竞争优势（我们的差异化）

**Product Planner 的独特价值：**

1. **🎯 深度集成 Claude Code**
   - 无缝融入开发工作流
   - 一条命令完成产品分析
   - 结果自动保存到项目中

2. **🧠 AI 驱动的智能分析**
   - 自动市场研究
   - 竞品自动发现和对比
   - 数据驱动的建议

3. **📋 系统化的方法论**
   - 专业 PM 框架
   - 结构化输出
   - 可复用的模板

4. **⚡ 为开发者优化**
   - 技术人员易理解的语言
   - 快速（分钟级完成分析）
   - 轻量级，零学习成本

5. **🔄 持续迭代支持**
   - 不只是一次性分析
   - 支持版本规划
   - 路线图管理

**护城河：**
- ✅ First mover advantage in Claude Code ecosystem
- ✅ 与代码开发深度集成
- ✅ AI 能力壁垒
- ✅ 开源社区效应

### 3.3 MVP 定义

**MVP 核心功能：**

**命令：`/product-plan [项目描述]`**

**工作流程（5个阶段）：**

#### Phase 1: 项目理解 🎯
- 收集项目基本信息
- 澄清不明确的需求
- 确认目标和约束

#### Phase 2: 市场研究 📊
- 市场规模和趋势分析
- 目标用户识别
- 竞品发现和对比
- **输出：** 市场分析报告

#### Phase 3: 产品定义 💎
- 核心价值主张
- 用户画像和痛点
- MVP 功能定义
- 功能优先级矩阵
- **输出：** 产品定义文档

#### Phase 4: 路线图规划 🗺️
- v1.0 MVP 范围
- v2.0+ 演进路径
- 技术可行性评估
- 资源和时间估算
- **输出：** 产品路线图

#### Phase 5: 成功指标与策略 📈
- 关键成功指标（KPI）
- Go-to-market 策略
- 推广渠道建议
- 风险评估
- **输出：** GTM 策略和 KPI

**最终交付物：**
- 📄 完整的 PRD（Product Requirements Document）
- 📊 可视化的功能矩阵和路线图
- 📈 成功指标追踪模板
- 🚀 发布计划清单

### 3.4 非功能需求

**性能：**
- ⏱️ 完整分析在 5-10 分钟内完成
- 🔄 支持增量更新（已有项目）

**质量：**
- ✅ 输出专业、结构化
- 📚 提供参考依据和来源
- 💡 可操作的建议

**用户体验：**
- 🎨 清晰的文档格式
- 📝 Markdown 格式输出
- 🔗 集成到项目目录

**扩展性：**
- 🧩 模块化设计
- 🔧 支持自定义分析维度
- 📦 可导出到其他工具
